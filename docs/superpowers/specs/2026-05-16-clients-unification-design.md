# Унификация раздела «Клиенты» — design spec

**Дата:** 2026-05-16
**Статус:** черновик, согласование с владельцем продукта (Mirzahit)
**Контекст:** на проде v289. Текущая фрагментация (Внедрение / Действующие / Ключевые / Очередь интеграции — 4 экрана работают с одним и тем же клиентом, дублируя данные и логику) разваливает CRM-архитектуру.

---

## 1. Цель

Один раздел **«Клиенты»** в сайдбаре. Внутри — табы-views одной унифицированной базы. Карточка клиента — единственная и адаптивная. Тех.поддержка остаётся отдельным управленческим дашбордом.

**Зачем:** взросление продукта до уровня CRM, удобство для пользователя (один источник правды, поиск работает один раз), снижение шума в сайдбаре с ~22 пунктов до ~13.

## 2. Унифицированная модель клиента

Единая запись `Client` содержит ВСЕ известные нам поля:

| Поле | Тип | Откуда сейчас | Заполняется |
|---|---|---|---|
| `id` | string | новое — slug из host или uuid | при создании |
| `name` | string | clients-kz.json `name` или kbCards `client` | при создании |
| `host` | string | clients-kz.json `host` | при создании |
| `status` | enum: `Внедрение` \| `Активный` \| `Расторгнут` | вычисляется из наличия в kbCards / payments | автоматически + ручной override. `Лид` добавляется позже когда появится AmoCRM webhook (см. §9). |
| `stage` | enum: `В очереди` \| `Взят в работу` \| `Настройка сервера` \| `Обучение полевых` \| `Обучения офисных` \| `Обучение Руководства` \| `Активация` | kbCards `stage` | актуально только если `status=Внедрение` |
| `segment` | enum: `A+` \| `A` \| `B` \| `C` \| `D` | вычисляется из `monthlyQty` | автоматически |
| `risk` | enum: `high` \| `med` \| `low` \| `норма` | вычисляется (см. § 4) | автоматически |
| `monthlyQty` | number | clients-kz.json `qty` (snapshot) | синхронизация |
| `monthlySum` | number | clients-kz.json `sum` (snapshot) | синхронизация |
| `manager_sales` | string | kbCards `manager` | при создании сделки |
| `operator` | string | kbCards `operator` | назначается админом |
| `activDate` | date | kbCards `activDate` | при переходе в `status=Активный` |
| `period` | number (мес) | вычисляется из payments (`amount / monthlySum`) или kbCards | при оплате |
| `lastPayDate` | date | агрегация payments по host-matching | автоматически |
| `nextExpectedPay` | date | `lastPayDate + period` | автоматически |
| `payCount` | number | агрегация | автоматически |
| `totalAmount` | number | агрегация | автоматически |
| `churnReason` | string \| null | новое поле | вручную при `status=Расторгнут` |
| `churnDate` | date \| null | новое поле | вручную при `status=Расторгнут` |
| `tags` | string[] | ручные теги | вручную |
| `integrationStatus` | enum: `не нужна` \| `нужна` \| `настроена` | новое поле, из текущей «Очереди интеграции» | админом / оператором |
| `comments` | array of {date, author, text} | kbCards comments | вручную |
| `tasks` | array | kbCards tasks | вручную |
| `files` | array | заглушка (Drive integration отложен) | будет позже |

### 2.1 Источники данных и где живёт единая модель

**Решение:** на первом этапе **не делаем новую Google Sheet**. Собираем `Client` on-the-fly из существующих источников при загрузке приложения:

- `clients-kz.json` (master список с qty/sum) — это активные / расторгнутые
- `kbCards` (Apps Script sheet канбана) — это лиды / внедрение
- `payments` (Apps Script sheet оплат) — для lastPayDate, period, totalAmount
- localStorage (`sd_client_overrides`) — ручные правки (status, churnReason, integrationStatus, tags)

Функция `buildUnifiedClients()` возвращает массив `Client[]`. Кешируется в `window.UNIFIED_CLIENTS`, инвалидируется при изменении любого источника.

**Why on-the-fly:** не ломает Apps Script, не требует миграции данных, можно откатить за минуты. Если архитектура себя оправдает (3-6 месяцев), вторым этапом мигрируем на единую sheet `Clients_v2`.

**How to apply:** все view'ы внутри «Клиенты» читают из `UNIFIED_CLIENTS`, фильтруют и сортируют. Источниковые объекты (`kbCards`, `payments`, etc.) остаются нетронутыми — мы их только агрегируем.

## 3. Структура раздела «Клиенты»

### 3.1 Сайдбар

В сайдбаре появляется один пункт **«Клиенты»** с иконкой пользователя. Группа «Клиенты» (где сейчас 5 пунктов: Ключевые/Очередь/Внедрение/Календарь/Тех.поддержка/Действующие) распадается:

- Календарь → переезжает в группу «Команда»
- Тех.поддержка → solo-кнопка отдельно (это дашборд, не клиенты)
- Остальные 4 (Ключевые / Очередь / Внедрение / Действующие) → исчезают, поглощаются «Клиенты»

### 3.2 Внутри «Клиенты»

Шапка раздела: переключатель табов, поиск, фильтры, кнопка «+ новый клиент».

**Табы:**

| Таб | Дефолтный вид | Фильтр | Источник цифр в счётчике |
|---|---|---|---|
| Все | Таблица | без фильтра | count(UNIFIED_CLIENTS) |
| Внедрение | Канбан | `status === 'Внедрение'` | count при фильтре |
| Действующие | Таблица | `status === 'Активный'` | count при фильтре |
| В риске | Таблица (сортировка по Δсумма asc) | `risk !== 'норма'` | count при фильтре |
| Ключевые | Карточки или таблица | `segment ∈ {A+, A}` | count при фильтре |
| Расторгнутые | Таблица | `status === 'Расторгнут'` | count при фильтре |

Все табы оперируют **одной таблицей-моделью**. У каждого таба настроен дефолтный вид (Таблица/Канбан/Карточки), но пользователь может переключить.

### 3.3 Виды

- **Таблица** — общая для всех табов, набор колонок отличается:
  - Все: Клиент / Статус / Сегмент / ₸/мес / Лиц. / Оператор / Менеджер
  - Внедрение: Клиент / Этап / Дней на этапе / Оператор / Менеджер / Сумма сделки
  - Действующие: Клиент / Лиц. / ₸/мес / Δ сумма / Δ лиц / Период / Следующая оплата / Оператор / Риск
  - В риске: те же что и Действующие, но отсортированы по Δ сумма asc
  - Ключевые: Клиент / Сегмент / ₸/мес / Δ за месяц / Δ за год / Оператор / Менеджер
  - Расторгнутые: Клиент / Дата ухода / Причина / Был у / Сумма за всё время / Менеджер
- **Канбан** — только для Внедрения и фильтров где есть `stage`. Drag-n-drop сохраняется (как сейчас в текущем канбане).
- **Карточки** — режим «сетки» для Ключевых и опционально для остальных.

### 3.4 Поиск, фильтры, sorting

- Глобальный поиск по `name`, `host`, `manager_sales`, `operator` — работает во всех табах.
- Дополнительные фильтры: оператор, менеджер продаж, сегмент, риск, теги.
- Сохраняемые view (стрейч): «Мои клиенты Самата в риске и >50K» как именованный пресет в localStorage.

### 3.5 Таб «Внедрение» — дополнительный фильтр «Нужна интеграция»

В текущей «Очереди интеграции» — это список клиентов с флагом «нужна настройка API». В новой архитектуре это становится **чипом-фильтром** в шапке таба Внедрение: `[ Все | Нужна интеграция (5) ]`. Один клик переключает.

Действия из текущей очереди («отметить как настроенную», «скопировать ссылку», «открыть карточку») переезжают в карточку клиента в блок «Интеграция».

## 4. Расчёт `risk` и `Δ` (амортизация платежей)

### 4.1 Амортизация

Для каждого платежа из `payments`:
- `amortStart` = `activationDate || date`
- `amortPeriod` = `period` (из платежа) или `Math.max(1, Math.round(amount / monthlySum))` если поле пустое
- Распределяем `amount / amortPeriod` ₸ и `qty / amortPeriod` лицензий на каждый из `amortPeriod` месяцев начиная с `amortStart`

Получаем таблицу `monthlyFact[clientId][YYYY-MM] = { sum, qty }`.

### 4.2 Расчёт риска

Для каждого клиента:
- `factThis` = `monthlyFact[id][currentMonth]?.sum || 0`
- `factPrev` = `monthlyFact[id][prevMonth]?.sum || 0`
- `qtyThis`, `qtyPrev` — аналогично

```
if factPrev === 0 && factThis === 0:
  # клиент не платил оба месяца — риск ставим по «просрочке» (§4.3),
  # если просрочки нет — норма (видимо новый, ещё не вошёл в платёжный цикл)
  risk = overdueDays > 7 ? 'high' : 'норма'
elif factPrev === 0 && factThis > 0:  risk = 'low' (только что начал платить)
elif factPrev > 0 && factThis === 0:  risk = 'high' (полностью прекратил)
else:
  deltaPctSum = (factThis - factPrev) / factPrev
  deltaQty = qtyThis - qtyPrev
  if deltaPctSum < -0.3 OR deltaQty < 0: risk = 'high'
  elif deltaPctSum < -0.1:               risk = 'med'
  else:                                  risk = 'низкий / норма'
```

### 4.3 «Просрочка» как параллельный сигнал

Не входит в `risk`, но показывается в строке клиента отдельным маркером.

- `expectedNext` = `lastPayDate + period` (последний платёж + его период)
- Если `today > expectedNext + 7 дней` и нет нового платежа → флажок «просрочка N дней»

Это жёлтый маркер у имени клиента, влияет на сортировку «в риске» (просроченные выше).

### 4.4 Рейтинг операторов в табе «В риске»

Над таблицей клиентов в табе В риске — компактная таблица операторов:

| Оператор | Клиентов | Падают | Δ сумма | Δ лицензии |

Сортировка по `|Δ сумма|` desc — у кого худшее падение, тот сверху. Клик по строке оператора = фильтр на список ниже.

## 5. Карточка клиента — адаптивная

Структура карточки (3 колонки, как сейчас):
- Левая: статичные данные (имя, статус, сумма, оператор, менеджер, etc.)
- Центральная: контент, зависит от статуса
- Правая: лента активности

### 5.1 Центральная колонка по статусам

| Статус | Что в центре |
|---|---|
| Внедрение | Этапы 1–7 + чек-листы текущего этапа (как сейчас) + блок «Интеграция» (нужна/настроена) |
| Активный | График оплат по месяцам (амортизированный), таблица лицензий по месяцам, история коммуникаций, блок «Риск и почему» |
| Расторгнут | Причина ухода, дата, последние коммуникации, статистика «был у нас X месяцев, заплатил Y ₸» |

`Лид` появляется позже (см. §9), тогда добавим четвёртый кейс.

### 5.2 Действия в карточке

Кнопка «Изменить статус» вверху карточки → выпадающее меню с переходами:
- Внедрение → Активный (ставит `activDate = today`, переводит из канбана)
- Активный → Расторгнут (просит указать причину + дату, выставляет `churnDate`)
- Расторгнут → Активный (восстановление)
- (после появления Лида) Лид → Внедрение — создаёт kbCard в «В очереди»

История смены статусов пишется в localStorage `sd_status_history` + отображается в ленте активности.

## 6. Что не делаем сейчас (out of scope)

- Миграция на единую Google Sheet `Clients_v2` — этап 2, после 3-6 месяцев работы on-the-fly модели.
- Drive-интеграция для файлов в карточке — отложено.
- AmoCRM webhook (новая сделка → автокарточка) — отложено.
- 2FA, активные сессии, history of logins — это для Профиля, не для Клиентов.
- Email/Telegram уведомления оператору при появлении нового высокого риска — этап 3, после Telegram-бота.
- ML-прогноз churn — этап будущего.
- Сверка с 1С — отдельный проект.

## 7. Технические решения

- **Совместимость:** старые URL'ы (`#view-kanban`, `#view-keyclients`, `#view-activeclients`, `#view-integrations`) → редирект на `#view-clients?tab=...` через `showView()`.
- **Permissions:** новый ключ `view_clients` для всех ролей, которые сейчас имеют любой из `view_kanban` / `view_keyclients` / `view_activeclients` / `view_integrations`.
- **Кэширование:** `UNIFIED_CLIENTS` пересчитывается при изменении `payments`, `kbCards`, `clients-kz.json` или ручных override. Listener-паттерн или явный invalidate-вызов после CRUD.
- **Версионирование:** релизы v290+ — этапами (см. §8).

## 9. Открытые вопросы

- Где хранить ручные правки `status / churnReason / churnDate / integrationStatus / tags`? Сейчас предполагаем localStorage `sd_client_overrides`. Это значит — невидимы между устройствами. **Принять решение перед этапом 2**: либо так и оставляем (для MVP), либо пишем endpoint в Apps Script и храним в новой sheet `Client_Overrides`.
- Что делать с клиентами которые есть в `kbCards` но НЕ в `clients-kz.json`? Они после активации должны попасть в clients-kz.json — это сейчас ручная процедура. В новой архитектуре это надо автоматизировать (через Apps Script триггер? через кнопку «синхронизировать список» в админке?). **Принять решение перед этапом 3**.
- «Лид» как статус — есть ли он у нас вообще? Если воронка начинается с «оплачено», то «Лид» это будущая сущность (когда сделаем AmoCRM webhook). Сейчас можно опустить из enum и добавить позже.

## 10. Метрики успеха

- ≥ 50% уменьшение времени на поиск клиента (с 3-5 кликов до 1-2)
- Один источник правды — невозможно увидеть разные `monthlySum` у одного клиента в разных экранах
- Сайдбар сокращён с 22 пунктов до ≤14
- 0 регрессий — все текущие функции (drag-n-drop канбана, фильтры, чек-листы, активность) работают
