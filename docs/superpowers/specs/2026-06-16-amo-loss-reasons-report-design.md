# Отчёт «Причины отказа» из amoCRM

Дата: 2026-06-16. Статус: одобрено, в реализацию (v629).

## Цель
Дашборд-отчёт по причинам отказа клиентов: по каким причинам и на какую сумму теряются сделки. Источник — нативное поле amoCRM «Причина отказа» (`loss_reason`).

## Решения (из брейншторма)
- Источник: штатное `loss_reason` amo (не теги/кастомные поля).
- Метрика: и количество сделок, и сумма (бюджет проигранных сделок) по каждой причине + доля.
- Размещение: карточка в разделе «Отчёты» (`view-reports`), открывается через `reportGenerate('loss_reasons')`.
- Период: переключатель внутри отчёта — Месяц / Квартал / Год.
- Доп.разрезы: дрилл-даун (клик по причине → список проигранных сделок), отдельная таблица разбивки по менеджерам, горизонтальные бары топ-причин.
- Воронка: основная «Лиды» (как существующий `getFunnel`). Разбивка по воронкам — НЕ делаем.
- Страна: KZ/KG через `currentCountry` (как остальные amo-вызовы).

## Архитектура
Один backend-вызов возвращает и агрегат, и компактный список сделок; фронт строит все срезы (таблица, бары, по менеджерам, дрилл-даун) из одного ответа — без повторных запросов к amo.

### Backend — `action=loss_reasons` в `api/amo.js`
Вход: `country`, `period` (this_month | quarter | year), опц. `pipeline_id`.
Шаги (переиспользуем `amoFetch`, `getPipelines`, `isLoss`, пагинацию):
1. Пайплайн «Лиды» (как в `getFunnel`); определяем статусы-потери: `status.id === 143` ИЛИ `isLoss(name)`.
2. Каталог причин: `GET /leads/loss_reasons` → `id → название`.
3. Пользователи: `GET /users` → `id → имя менеджера`.
4. Период → unix from/to (this_month: 1-е число тек. месяца … now; quarter: начало квартала; year: 1 янв … now).
5. Пагинация лидов: `filter[statuses][i][pipeline_id]+[status_id]` по статусам-потерям, `filter[closed_at][from/to]`, `with=loss_reason`, `limit=250`, до N страниц (флаг truncated при упоре в лимит).
6. По каждому лиду: причина из `_embedded.loss_reason[0].name` (пусто → «Причина не указана»), `amount=price`, `manager=users[responsible_user_id]`, `company=name`, `date=closed_at`, `lead_id`.
7. Возврат: `{ period:{label,from,to}, currency, pipeline:{id,name}, total:{count,sum}, reasons:[{name,count,sum}], deals:[{company,amount,manager,reason,date,lead_id}], truncated }`. Кэш по `(country, period)` (как `_amoFunnelCache`).

### Frontend — `reportGenerate('loss_reasons')` (асинхронный)
- Карточка «Причины отказа» в `view-reports` (перед «Добавить отчёт»).
- Поток: «загрузка…» → `authFetch('/api/amo?action=loss_reasons&country=&period=')` → рендер в `#report-result-body`.
- Состояние периода: `window._lossReasonsPeriod` (по умолчанию this_month); кнопки-переключатели перерисовывают.
- Рендер: переключатель периода → 2 плашки (всего сделок / сумма) → горизонтальные бары топ-причин по сумме → таблица причин (причина, кол-во, сумма, доля) → таблица по менеджерам (из `deals`) → клик по строке причины раскрывает её сделки (фильтр `deals` по reason, без нового запроса).
- Все строки из amo (company, reason, manager) — через `sanitize()` (учитываем XSS-аудит).
- Деньги через `fmtFull`/`fmt` + `_CUR`.

## Edge cases / риски
- Нерегулярно заполненная причина → бакет «Причина не указана» (отчёт честно подсветит качество данных).
- Пустой период → «За период проигранных сделок нет».
- `truncated` → подпись «показаны первые N сделок» (без молчаливого среза).
- amo env не настроен / ошибка → понятное сообщение об ошибке в теле отчёта.
- Версия: бамп title + sw CACHE_NAME (v629).

## Вне scope
Разбивка по воронкам; запись/редактирование причин (отчёт только на чтение); экспорт.
